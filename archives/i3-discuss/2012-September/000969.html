On Thu, Sep 27, 2012 at 3:57 PM, Michael Stapelberg <michael at i3wm.org>wrote:

> Hi darkraven,
>
> DR <drdarkraven at gmail.com> writes:
> > I've read your criteria at http://www.featurequest.org/FRQ$CodeReview.
> From
> > what I've read I think GitHub is good enough for you.
> No. GitHub (while being a cool company with a good product) is
> proprietary. I cannot host their review software on my own
> server, thus using GitHub for code reviews is a vendor lock-in.

I won't try to persuade you to use GitHub, but I think some of your points
are quite arguable.

>
> I want to avoid such a vendor lock-in for several reasons:
>
> 1) Building free software should not depend on closed-source
>    software. I’m not saying this in a _very_ religios fashion such as
>    rms (who’d argue that my server’s BIOS is probably unfree), but it
>    seems reasonable to avoid closed-source if we can reasonably live
>    without it. We’ve seen what happened with BitKeeper and the Linux
>    kernel.
>
I'm quite new on linux, and BitKeeper is long gone before I even heard of
linux, so I could be wrong on this. But for what I understand, it's
religious people like RMS who forced linux and BitKeeper to part ways, not
BitKeeper's being proprietary limited the development of kernel.

>
> 2) If we want to improve the code review workflow in some detail, we
>    cannot do it. Unless GitHub is friendly enough to do it, if it makes
>    sense for them at all. This is the major reason why I switched to a
>    FOSS operating system and software. Similarly, they have the option
>    of modifying the software as they see fit, without asking us first
>    and without an alternative for us.
>
Imagine I switched to linux distributions from Windows, and change it to
look&behave exactly the same as Windows, then why would I make the change?
If you not really have a better idea, why not just use the existing
solution, even if it's closed?
I don't know you, maybe you have a definitely better idea of how a code
reviewing system should be than GitHub. But if not, why not using GitHub
for now, and when you eventually feel the itch to make changes, you could
start making your own. And by that time, what you make will definitely  be
better.
Oops, this sounds like persuading. Anyway, I'm just using GitHub as an
example to make my point.

>
> 3) GitHub requires an account to submit a pull request. This is
>    reasonable from GitHub’s point of view but unacceptable from ours. No
>    account creation barriers to reporting bugs and sending improvements
>    has been one of our principles since the very beginning of the
>    project.
>
This could be a problem. But GitHub is not dedicated to i3, so people who
want to contribute might "happens" to have a GitHub account.

>
> 4) With using GitHub for reviews, we effectively have to use GitHub for
>    our git repository. Otherwise we’d have to mirror it, but having two
>    copies which might be out of sync is stupid, so we’d have to push to
>    both locations. GitHub (in my daily life) is slow. And sometimes it’s
>    down. My own server is much more reliable for our little
>    project. Also, it seems wise to avoid a central point-of-failure and
>    encourage a distributed system (many git servers instead of every
>    FOSS project on GitHub).
>
My perception of "distribution" is a bit different as yours. I think being
distributed means every one who clone the repo have complete history of the
repo. So have one *more* clone on GitHub won't harm this kind of
distribution at all.
There are no two copies of one repo, there are two different branches which
could pull from each other. I think out-of-sync would be too big a problem.
And the network problem is a matter of view point. GitHub might be slow for
you, but faster for someone else, and your server might be fast for you,
but slow for someone else.


> --
>
If my points sound stupid, point that out and help cure my stupidity,
thanks!

>  Best regards,
> Michael
>
Best regards.
