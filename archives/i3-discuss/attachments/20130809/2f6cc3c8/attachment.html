<div dir="ltr"><div>Hi,</div><div><br></div><div>I read the following ticket listed as major priority: <a href="http://bugs.i3wm.org/report/ticket/662">http://bugs.i3wm.org/report/ticket/662</a></div><div><br></div><div>First it is quite old, but that's not the point.</div>

<div><br></div><div>Second I doubt that any of these features would be really useful.</div><div>The "time out" is clearly a bad idea. Letting the computer deciding itself to reset a mode which has been enabled by the user can introduce some sort of desynchronization between the user and the computer state. Depending on his mood one can perceive the time differently so a fixed time out value can be sometime confusing. It ends up that there is no good fixed value for such a time out.</div>

<div>The "bindelse" is less "evil" but it does not address the issue, for instance, if I press the wrong key, my reflex is to immediately correct this move and press the good key. In the case of "bindelse" the first wrong key would reset the mode and the effect of the next good input is undefined.</div>

<div><br></div><div>So I think that this post has one solution: <a href="http://infra.in.zekjur.net/archives/i3-discuss/2011-January/000180.html">http://infra.in.zekjur.net/archives/i3-discuss/2011-January/000180.html</a></div>

<div><br></div><div>The reply is clear: "<span style="color:rgb(0,0,0);white-space:pre-wrap">This appears to be the silver bullet!"</span></div><div><font color="#000000"><span style="white-space:pre-wrap">This already working solution allows a fine-grained control of which inputs reset the mode.</span></font></div>

<div><font color="#000000"><span style="white-space:pre-wrap"><br></span></font></div><div><font color="#000000"><span style="white-space:pre-wrap">The next reply about sticky idea is very clever and reflect what I experienced when playing with binding modes. It happens that if one key should reset the mode then there is a high chance that the whole key binding group should do it too.</span></font></div>
