<div>Hey guys,</div><div>I am going to start writing a gesture recognizer for utilizing pointer devices (singe/multi touch) on linux as a front end to window managers. I wanted to hear your opinions on it before actually starting the development.</div>
<div> </div><div>I'm thinking of a layered structure like this</div><div>__gesture recognizer</div><div>__gesture recognizer interface for window manager</div><div>__window-manager</div><div>So, simply, a gesture recognizer is gonna be an app that prints an identifier when a gesture is recognized</div>
<div>The middle layer is going to read those identifiers, and send commands to the window manager</div><div>And the window manager is... the window manager.</div><div> </div><div>I'd like the user to be able to change to a different gesture recognizer without having to do too much tweaking. I'd like it to be usable on all the window managers.</div>
<div> </div><div>There are obvious issues though</div><div>Say "gr1","gr2","gr*" are gesture recognizers. They print an identifier like "close-win","switch-next-workspace" etc.</div>
<div>"i3wm-gr-interface", "blackbox-gr-interface" etc are various middle layer applications</div><div>"i3wm", "blackbox" etc are the window managers</div><div> </div><div>consider the middle layer:</div>
<div>* i3wm-gr-interface will likely be implementing i3wm-ipc-protocol.</div><div>* blackbox-gr-interface however will implement xlib</div><div>And sadly, there are not many window managers like i3 that's easy to communicate with. For example when I need to list available workspaces in a window manager other than i3, I will have to implement workspaces myself, because there is no means of learning about them from blackbox. When this is the case, it's better to embed the gesture recognizer within the window manager.</div>
<div> </div><div>Should I just stick with writing a gesture recognizer for i3wm only? This way, the top and the middle layers can be put together. This will be my route of choice, unless I learn about a way window managers can be communicated with.</div>
