> (3) I just don't grasp the concept of a container in i3. It seems to<br>

<br>
</div>Mathematically it is pretty simple. Containers are nodes in a tree,<br>
whose root the X-root-window is. Children of this root level there are<br>
individual outputs, children of the outputs are the workspaces, whose<br>
children in turn are the container-trees containing the windows you see.<br>
<br>
A container can either be an inner node containing more containers (I<br>
think this is called a „split-container“ in the documentation), or it<br>
can be a leaf, containing a window.<br>
<br>
The nodes are not all equal, leafs contain windows and near the top of<br>
the hierarchy there has to be special treatment (to use the more<br>
user-friendly abstractions of outputs and workspaces), but mostly this<br>
should be the way to think of it.<br>
<div class="im"><br>
> (2) If I press mod+v and make some windows, are they all in the same<br>
> container? If I now press mod+h and make some more windows, are they<br>
> in a different container?<br>
<br>
</div>Yes and no. If by „window“ you think of „a leaf node in the container<br>
tree“, then yes.<br>
<br>
If you create some windows (for example pressing Mod+Enter to<br>
open some terminals), then each one is put in an individual container.<br>
These containers are then put in a big container. If you then press<br>
mod+h, the currently focused node in the tree (which would be a leaf in<br>
most cases, so a container having a window inside it) is replaced by a<br>
new container, with the old content as a subtree. If you then open some<br>
windows, they are new leafs which are put in that new container.<br>
<br>
So I think the solution here is to stop thinking about windows as<br>
entities in the container-tree, because they're really not. The<br>
container-tree is an abstraction used by i3 and leaf-nodes in this tree<br>
have a window-property attached to them.<br>

<br>
      Workspace<br>
    /  |  |  | \<br>

<br>
