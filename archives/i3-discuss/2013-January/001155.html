1. I dont think there is a option for auto toggle between splitv and
splith. maybe you mean default_orientation auto? but it is statically
determined by the screen size in the beginning. (p.s. maybe awesome
wm?)

2. I forget to talk about the benefits in my last post. let's analyze a bit:
* when we continuously use mod+ return, both stacking and tabbed
result in a readable window size. however splitvh does not. I know
someone mentioned that if the screen is large enough, you can actually
split 3 or 4 times. but the property of complete binary tree is that
all windows (except those in the deepest level) should have about the
same window size. if you keep splitting more windows, the property
ensures that all windows do not get ridiculously tall or fat. (if you
still want to split 3, 4 times, you can manually move the new windows.
but i dont think cbtree is less readable.)

* as mentioned, this actually benefits people who have smaller screen
(12 in x 8 in) like me. I cannot do splitv because this makes windows
too fat, one splith is ok but more than that makes the windows too
thin. because i have to move things around, this is
counter-productive. (I now stay with tabbed, but by default it is hard
to look at 2 windows together)

* split v and split h comment is still there. but by default splitv
splith orientation are not that useful (for reason above).  The
purpose of default orientation is that it reduces work for users, so
cbtree model is just about right.

>
>
> when we insert a new node (windows) for splith
>            h
>        v    v
>      h  3 2  4
>     1 5

>
>      h
